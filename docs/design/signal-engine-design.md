## Problem Statement

Taiwan stock analysis tools (CMoney, XQ) provide price/volume data and basic institutional flows, but lack deep broker-level chip synthesis. The gap: no tool automatically identifies which broker branches are 隔日沖 shops vs. wave traders (波段贏家) vs. regional institutional proxies (地緣券商), and no tool fuses that label data with momentum signals to generate structured entry/exit decisions. A trader who knows what to look for must spend hours manually correlating data that could be computed in seconds.

## Demand Evidence

- User is a self-directed Taiwan stock trader currently paying for CMoney/XQ
- Both tools are actively used (paying habit exists), but neither provides broker-level chip synthesis
- The gap is felt daily: missing the "authenticity" check on whether a price move has real accumulation or is a next-day reversal setup
- Existing demand signal: user's willingness to build a complex system rather than switch to a different existing tool confirms the gap is real and currently unserved

## Status Quo

Current workflow: CMoney/XQ for price data + manual checking of TWSE website for broker-branch daily data. Analysis takes 1-2 hours post-market per watchlist. Key friction points:
- No automated broker label lookup (is this branch historically a 隔日沖 shop or a wave trader?)
- No integrated Triple Confirmation scoring (must mentally combine VWAP position, chip concentration, and Volume Profile POC)
- No structured output — conclusions stored as notes or held in memory

## Target User & Narrowest Wedge

**Wedge target:** Self-directed Taiwan stock traders who already do chip analysis manually — users who know what 籌碼集中度 and 分點 mean and currently do the analysis by hand. NOT passive retail investors who "want signals without skill" (that segment has <10% week-4 retention without social proof/portfolio tracking features).

**Narrowest viable version:** A daily post-market CLI tool that:
1. Ingests T+1 分點 data via FinMind API
2. Labels broker branches (隔日沖 / 波段贏家 / 地緣券商 / 代操官股) from historical behavioral fingerprints
3. Scores the Triple Confirmation for a watchlist of tickers
4. Outputs a structured JSON signal + human-readable summary

**This is something one self-directed trader would pay for this week.**

## Constraints

- **Data:** No real-time public 分點 data. All broker-level data is T+1 (end-of-day). Real-time requires a brokerage API partnership (Fubon Neo, Sinopac) — deferred to Phase 4.
- **Infrastructure:** Kafka/Redis premature for MVP. FinMind polling + PostgreSQL is sufficient until the user base and tick frequency warrant it.
- **Agent orchestration:** LangGraph adds latency and failure modes over a deterministic rule engine. The Triple Confirmation logic is if/else, not reasoning. LLM used only for natural language output.
- **Segment:** "Without skill" retail investors are a high-churn, high-risk segment for early stage. Start with traders who can act on structured signals.

## Premises

1. No real-time 分點 data without a broker API partnership — intraday module is Phase 4
2. Kafka/Redis/TimescaleDB/ChromaDB is premature for MVP — FinMind + PostgreSQL is sufficient
3. LangGraph multi-agent orchestration adds complexity over a Python rule engine for deterministic Triple Confirmation logic
4. "Without skill" retail users are the highest-churn, highest-risk first segment
5. The broker labeling database (classifying branch codes by behavioral fingerprint) is the actual competitive moat — not the agent architecture

## Recommended Approach

**Approach B: Agentic Signal Engine (Phased)** with explicit phase gates.

### Architecture (adjusted from original plan)

```
Infrastructure Layer:
  - FinMind API client (T+1 分點 data via TaiwanStockBrokerTradingStatement;
    OHLCV via TaiwanStockPriceAdj for all historical/backtest analysis — adjusted for dividends and splits;
    raw TaiwanStockPrice used only for live execution price reference at signal time)
  - PostgreSQL (chip history, broker labels, OHLCV, volume profiles)
  [Future Phase 4: WebSocket broker API for real-time tick data]

Domain Layer:
  - BrokerLabelClassifier: behavioral fingerprinting (隔日沖 / 波段贏家 / 地緣券商 / 代操官股)
  - TripleConfirmationEngine: VWAP + volume surge + chip concentration + POC check
    (includes risk deductions: 隔日沖 flag, momentum divergence — RiskAlertEngine is NOT a separate class)
  - VolumeProfileBuilder: 20-session daily OHLCV → POC + resistance nodes

Agentic Layer (Python classes, not LangGraph):
  - ChipDetectiveAgent: broker label lookup, concentration scoring, net_buyer_count_diff
  - StrategistAgent: aggregates OHLCV + ChipReport + VolumeProfile → confidence score + JSON output
  [Phase 2+: ScoutAgent — market anomaly scanner over user-defined watchlist]

Presentation Layer:
  - JSON output (schema above)
  - LLM call (Claude API) for natural language reasoning fields ONLY
  - CLI interface for daily post-market run: python -m taiwan_stock_agent --date 2026-03-24
```

### Key Design Decisions

1. **LangGraph → Python classes:** The three agents (Scout, Chip Detective, Strategist) are Python classes with typed interfaces, not LangGraph nodes. This achieves the same logical separation without the dependency overhead.

   **Agent interface contracts:**
   ```python
   # ScoutAgent (deferred to Phase 2 — not needed for 30-ticker watchlist in Phase 1)
   # Phase 2+: Input: date, watchlist: list[str]
   # Output: list[AnomalySignal(ticker, trigger_type, magnitude)]
   # Data source: FinMind TaiwanStockPrice (daily OHLCV)

   # ChipDetectiveAgent
   # Input: ticker: str, date: date (T+1 settlement), broker_label_db: BrokerLabelRepository
   # Output: ChipReport(
   #   top_buyers: list[BrokerWithLabel],        # top-15 branches ranked by buy volume
   #   concentration_top15: float,               # top-15 buy vol / total buy vol
   #   net_buyer_count_diff: int,                # see definition below
   #   risk_flags: list[str]                     # e.g. ['隔日沖_TOP3', '動能衰竭']
   # )
   # net_buyer_count_diff = sum over last 3 T+1 days of (distinct_buying_branches - distinct_selling_branches)
   # A POSITIVE value means more buyers than sellers by branch count (chips concentrating = smart money accumulating)
   # A branch that both buys and sells counts in BOTH sets for that day
   # Condition for concentration signal: net_buyer_count_diff > 0 (not < 0 as originally written)
   #
   # Reference SQL for net_buyer_count_diff:
   # WITH daily AS (
   #   SELECT trade_date,
   #          COUNT(DISTINCT CASE WHEN buy_volume > 0 THEN branch_code END) AS buy_branches,
   #          COUNT(DISTINCT CASE WHEN sell_volume > 0 THEN branch_code END) AS sell_branches
   #   FROM broker_trades
   #   WHERE ticker = %(ticker)s AND trade_date >= %(start_date)s AND trade_date <= %(end_date)s
   #   GROUP BY trade_date
   #   ORDER BY trade_date DESC
   #   LIMIT 3
   # )
   # SELECT SUM(buy_branches - sell_branches) AS net_buyer_count_diff FROM daily;

   # StrategistAgent
   # Input: ticker, ohlcv: DailyOHLCV, chip_report: ChipReport, volume_profile: VolumeProfile
   # Output: SignalOutput (see JSON schema below)
   ```

2. **Triple Confirmation formula (deterministic):**

   **Data availability note:** FinMind does not provide 1-minute intraday volume on free/standard plans. Phase 1-2 uses daily OHLCV only. The volume surge signal uses daily volume vs. 20-day average as a fallback.

   ```
   score = 0

   # Pillar 1: Momentum (0-40 pts)
   # Phase 1-3 (T+1 daily data):
   # vwap_5d = sum(close_i * volume_i for last 5 days) / sum(volume_i for last 5 days)
   # Note: daily VWAP = close (tautology with single bar), so 5d VWAP proxy used instead.
   if close > vwap_5d:                             score += 20  # price above 5-day volume-weighted avg
   if daily_volume > daily_20ma_volume * 1.5:      score += 20  # daily vol surge fallback
   # Phase 4+ (real-time, requires broker WebSocket):
   # if volume_1min > volume_20min_avg * 3.0:      score += 20  # 1-min surge (replaces daily fallback)

   # Pillar 2: Chip (0-40 pts)
   if net_buyer_count_diff > 0:                    score += 15  # chips concentrating: more buyer branches than seller branches
   if concentration_top15 > 0.35:                  score += 15  # top-15 branches >35% of total buy vol
   no_daytrade_in_top3 = all(b.label != '隔日沖' for b in top_3_buyers)
   if no_daytrade_in_top3:                         score += 10  # quality confirmation

   # Pillar 3: Space (0-20 pts)
   # Phase 1-3 proxy (daily OHLCV only — real VolumeProfile requires intraday data, deferred to Phase 4):
   # Use 20-day high as price-space confirmation proxy.
   # poc_price proxy = 20-day high. target_price = poc_price * 1.05 (5% above).
   if close > twenty_day_high * 0.99:              score += 20  # within 1% of or above 20-day high
   # Phase 4+: replace with real VolumeProfile (intraday tick data → POC + resistance nodes)

   # Risk deductions
   if not no_daytrade_in_top3:                     score -= 25  # 隔日沖 in top-3 flag
   # momentum_divergence_detected: close >= 5-day high AND (external_buy_ratio today < external_buy_ratio 3 days ago - 0.10)
   # external_buy_ratio = volume transacted at ask / total volume (外盤比); requires intraday data
   # Phase 1-3 fallback: skip divergence deduction (external_buy_ratio unavailable from FinMind T+1 data)
   if momentum_divergence_detected:                score -= 15  # price new high, external buy ratio declining

   confidence = max(0, min(100, score))
   ```

   **Action mapping:** `confidence >= 70 → LONG` | `confidence <= 30 → CAUTION` (avoid entry; note: short-selling requires margin eligibility not assumed here) | else → `WATCH`

   **Stop-loss note:** `stop_loss = T+0 closing price` (daily close — NOT intraday VWAP, which requires tick data unavailable from FinMind in Phase 1-3). This is an order-entry reference computed at post-market time. It does not account for gap-open scenarios. Traders must manually verify stop validity at market open next day.

3. **JSON output schema (field source annotated):**
   ```json
   {
     "ticker": "2330",           // deterministic
     "date": "2026-03-24",       // deterministic
     "action": "LONG",           // deterministic: confidence >= 70 → LONG, <= 30 → CAUTION, else WATCH
     "confidence": 88,           // deterministic (formula above)
     "reasoning": {
       "momentum": "...",        // LLM-generated (Claude API)
       "chip_analysis": "...",   // LLM-generated
       "risk_factors": "..."     // LLM-generated
     },
     "execution_plan": {
       "entry_bid_limit": 985,   // deterministic: close * 0.995 (lower bound, limit order)
       "entry_max_chase": 990,   // deterministic: close * 1.005 (upper bound, max chase)
       "stop_loss": 972,         // deterministic: T+0 closing VWAP (order-entry reference, not real-time)
       "target": 1015            // deterministic: nearest Volume Profile resistance node above POC
     }
   }
   ```

4. **Broker label database schema (PostgreSQL):**
   ```sql
   CREATE TABLE broker_labels (
     branch_code  VARCHAR(10) PRIMARY KEY,  -- e.g. '1480' (凱基台北)
     branch_name  VARCHAR(100),
     label        VARCHAR(20),              -- '隔日沖' | '波段贏家' | '地緣券商' | '代操官股' | 'unknown'
     reversal_rate FLOAT,                  -- P(next-day open-high → close-below-prev-close | branch in top-3)
     sample_count  INT,                    -- n trades used to compute rate
     last_updated  DATE,
     metadata     JSONB                    -- arbitrary features for future use
   );
   ```
   Classification rule: `label = '隔日沖'` if `reversal_rate > 0.60 AND sample_count >= 50`. Threshold is the pre-validation target from The Assignment. New branches default to `'unknown'` until sample_count reaches 50.

5. **ChromaDB/Pinecone deferred:** Broker labels live in PostgreSQL as above. Vector search adds no value until doing semantic retrieval over thousands of historical patterns — Phase 4+ territory.

6. **FinMind rate limits:** Free tier: 600 requests/day. A 2-year historical pull for 30 tickers × daily 分點 data ≈ ~21,900 rows, achievable in ~40 API calls (FinMind returns date-ranged batches). Free tier is sufficient for Phase 1 backfill. Phase 3 daily polling (30 tickers × 1 call/day) = 30 req/day, well within limits. Upgrade to paid plan ($5/month) if watchlist exceeds 200 tickers.

## Success Criteria

- **Pre-Phase-1 spike:** 隔日沖 branch reversal rate >60% (vs. ~35% unconditional baseline). Measurement: compute reversal_rate for ALL branches with ≥50 samples; confirm that branches widely known as 隔日沖 shops (by trader community consensus) cluster above 0.60, while unknown/random branches cluster near 0.35. This validates the metric before using it as the classifier criterion.
- **Phase 1:** Broker label classifier assigns labels to all branches with ≥50 samples. Validation: walk-forward test — use 2022-2023 data to set thresholds, compute reversal_rate on 2024-2025 hold-out, measure lift over unconditional baseline (target: labeled-as-隔日沖 branches show ≥1.5x the baseline reversal rate in hold-out period). F1 metric is not applicable here as there is no separate ground-truth label set — lift over baseline is the correct metric.
- **Phase 2:** Triple Confirmation filter on 30 backtested tickers shows positive expected value after transaction costs (assume 0.3% round-trip cost per trade, holding period 1-5 days). Backtest uses walk-forward validation (train on 2022-2024, test on 2025) to prevent look-ahead bias.
- **Phase 3:** CLI runs daily post-market in <60 seconds and produces actionable JSON + readable summary
- **Phase 4:** Real-time intraday signals with <5s latency from tick to decision

## The Assignment (Pre-Phase-1 Validation Spike)

Before writing any production code: **validate the broker label classifier hypothesis.**

This is a spike, not Phase 1. It takes ~2 hours and answers the most important question.

Pull 2 years of FinMind 分點 data for 5 well-known 隔日沖 broker branches (e.g., 凱基台北, 摩根大通). Measure:
- On days when these branches are top-3 buyers in a stock, what % of the time does the stock (a) open higher on D+1 AND (b) close below the previous day's close by end of D+1?
- **D+2 execution reality:** FinMind 分點 data is published T+1 night (Day D broker flows visible on Day D+1 evening). Earliest tradeable execution is **Day D+2 open**. The spike MUST also measure: what is the reversal rate at D+2 (not just D+1)? A pattern exploitable only on D+1 is a data latency artifact, not a tradeable signal.
- Baseline for comparison: unconditional P(open-high → close-below-prev) across all Taiwan stocks = approximately 30-35%. A signal requires >60% for top-隔日沖 branches **measured at D+2 execution**.
- Backtest methodology: use FinMind `TaiwanStockBrokerTradingStatement` (T+1 settlement), join with `TaiwanStockPriceAdj` (adjusted OHLCV, dividend/split-corrected). Holding period = 1 day. No transaction costs needed for this hypothesis test.
- Step zero: before analysis, confirm FinMind 分點 settlement dates align with OHLCV trade dates using 2330 on a known date (e.g., 2025-01-15). Misaligned join keys produce systematic false signals.

If the answer is >60%, the moat hypothesis is confirmed — build Phase 1.
If it's 40-60%, the signal is weak — revisit the classifier feature set before building.
If it's ~35% (baseline), broker labeling has no predictive power for this pattern — need a different hypothesis.

This one query tells you whether the system is worth building.

## Reviewer Concerns (Unresolved after 3 iterations)

1. **VolumeProfile resistance node bucketing undefined.** `target_price` is "nearest resistance node above POC" but the bucketing resolution and node identification algorithm are not specified. Engineers will need to define this — suggested: 1% price buckets, node = bucket with >5% of 20-session cumulative volume.

2. **`concentration_top15` edge case for thinly traded stocks.** If fewer than 15 branches are active for a ticker, top-15 = all branches and the threshold loses meaning. Suggested: skip concentration check if active_branches < 10; add `active_branch_count` to ChipReport output.

3. **Pre-phase spike data alignment.** FinMind 分點 settlement dates must be verified to align with OHLCV trade dates before running the spike. Run a single-stock sanity check first (known ticker, known date) to confirm the join key works.
